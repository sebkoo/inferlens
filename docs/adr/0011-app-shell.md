# ADR-0011: The app shell — a committed minimal project, and what invariant 5 forbids

- Status: Accepted — 2026-07-21
- Deciders: maintainer
- Relates to: [ADR-0002](0002-litert-distribution.md) (the one binary dependency and its pin —
  unchanged here), [ADR-0009](0009-document-store-scope.md) (the decision standard this one is
  held to), [ADR-0010](0010-remote-leg-scope.md) § Consequences (the composition this shell
  hosts), CLAUDE.md invariant 5 (precised by this decision), invariant 7 (what the shell finally
  feeds), invariant 8 (bootstrap precedes build), and the ladder's rung 37.

## Context

The composed app has been real since the composition rung:
[`InferlensApp.swift`](../../Sources/InferlensApp/InferlensApp.swift) composes the chain, the
ledger, the screen and the export, and it builds for the simulator inside every `test-clean` run.
What a pure-SPM `executableTarget` cannot produce is an app bundle: no `Info.plist`, no
signature, nothing a phone will install. ROADMAP's launchability raise (recorded at the rung-29
landing) named closing that gap an invariant-5-touching DECISION with an ADR's weight, not a
chore: invariant 5's wording was written before this fork existed, and the decision must either
fit inside it or amend it the recorded way. The measurement rungs are the forcing function —
on-device latency (rung 32), the method doc's numbers (rung 33) and the demo video (rung 36) all
require the app running on hardware.

## Decision 1 — the shell is a committed, hand-authored, minimal Xcode project at `App/`

`App/Inferlens.xcodeproj` wraps the existing package. Zero new tools; the standard signing and
run path. The raise's stated cost of this option — a diff-noisy project file — is contained by
construction rather than accepted:

- **Hand-authored and explicit.** One app target, three file references (the composition source,
  the `Bundle.module` shim, the `Models` folder), one local-package reference. Nothing
  Xcode-managed churns because nothing is Xcode-managed: signing lives in a git-ignored xcconfig,
  and user state (`xcuserdata/`) is git-ignored in the same commit, so opening the project does
  not dirty the tree.
- **Not at the repo root.** [`scripts/test-clean.sh:56`](../../scripts/test-clean.sh) resolves
  `-scheme Inferlens-Package` with no `-project` flag, against whatever container xcodebuild
  discovers in the working directory. A project at the root would enter that discovery, and the
  green bar's scheme resolution with it; under `App/` the shell is invisible to it. (Probed at
  the feat landing — see the execution notes in `docs/prompts/rung-37-app-shell.md`.)
- **Library products only.** The app target compiles `InferlensApp.swift` — referenced in place,
  it does not move — plus the shim, and links the same eight library products the package's
  executable target names. It never links the SPM executable target, and the package's
  `executableTarget` stays exactly where it is, so the composition keeps building, and keeps
  being verified, inside the package scheme the suite runs. The one drift surface this leaves is
  named: BUILD SETTINGS — the product list, the deployment target, the resource wiring — now
  exist in two places, pbxproj and `Package.swift`. The CODE does not: the composition file is
  single-source, compiled by both.
- **Resources ship the verified bytes.** The `Models` directory (staged by `make bootstrap`,
  ADR-0002's checksum discipline) enters the bundle as a FOLDER REFERENCE — copied verbatim, the
  same un-processed bytes the package's `.copy` ships. A processed resource would not be:
  Xcode's default coremlc compile of a `.mlmodel` rewrites the bytes `make bootstrap` verified,
  which is exactly what the package's `.copy` choice exists to prevent.
- **The shim is one honest line.** SPM synthesizes `Bundle.module` for the package build; an app
  target has no synthesized accessor, and its bundle IS the main bundle.
  `App/BundleModule.swift` maps the one onto the other and carries this reasoning as its comment,
  both build contexts named.
- **The scheme is shared.** Committed under `App/Inferlens.xcodeproj/xcshareddata/`, so a future
  CI device lane has a stable `-scheme Inferlens` name instead of a per-user autogenerated one.

## Decision 2 — invariant 5 is precised, not loosened

Invariant 5's target was always DEPENDENCY MANAGEMENT. "No CocoaPods. The build is pure SPM"
forbade a second dependency manager and an unpinned binary; it never had an app shell in view,
because this fork did not exist when it was written. The precised wording lands in CLAUDE.md in
the same commit as this file — the "exactly one" → "at most one" precedent for how a wording is
corrected without loosening it. What still fails review:

- a second dependency manager, in any form;
- an unpinned or checksum-less binary anywhere in the build;
- a package-resolution override inside the pbxproj — a pinned revision, a fork URL, a replaced
  package. Dependencies resolve through SPM against `Package.swift`; the pbxproj is an app-shell
  artifact with no say in them.

## Decision 3 — rung 37 owns the shell, and the raise's rung-36 attribution is corrected

The raise pointed at rung 36 as the rung that would hit the gap first. True as a PREDICTION —
the demo video cannot exist before an installed app — and wrong as OWNERSHIP: rung 36's ladder
line is the README's completion, and a `rung-36` tag on a shell commit would make the derived
status block claim work that has not happened. The rung-31 lint clause already on the ladder —
"every `rung-*` tag names a real ladder rung" — is the reason a 36-tag was refused; the lint
does not exist yet, and it just prevented its first violation. So the ladder gains rung 37, the
shell's own line, in the Measurement phase: the rung's stated reason for existing is that the
measurement rungs require an installable app, so that phase is its honest home. The raise itself
is corrected the recorded-correction way at this rung's docs commits — prediction and ownership
both named, nothing silently reworded.

## Signing — the maintainer's, by construction

The committed `App/Inferlens.xcconfig` ends with `#include? "Signing.local.xcconfig"` — an
optional include, so a clean clone builds unsigned (the simulator; `CODE_SIGNING_ALLOWED=NO` for
the device slice) and the personal team exists only on the machine that owns it. Creating it is
one line:

```
echo 'DEVELOPMENT_TEAM = <your team id>' > App/Signing.local.xcconfig
```

The file is git-ignored in the same commit that ships the project. The session prepares
everything up to that line; the install and the run are the human's — a checkpoint, not a
workaround.

## Bundle identity — chosen as permanent

`io.github.sebkoo.inferlens`. A bundle identifier is effectively permanent the moment the App
Store path is ever taken: the app record binds to it, and changing it later means a new app. The
identifier is anchored to the repo's real home, which stays honest for as long as the repo lives
at that address — the same lifetime every doc link in this repo already bets on. The xcconfig
comment says so, because the permanence was chosen, not stumbled into.

## The outcomes that were available and were not taken (the ADR-0009 standard)

**A generated project — XcodeGen or tuist, spec committed, project git-ignored.** What it alone
provides: the spec as single source of truth, and diffs that read as intent. Its cost: a new
tool the fails-closed bootstrap must pin and fetch — supply-chain surface bought to solve a
churn problem Decision 1 contains by other means. What unblocks it: the hand-authored pbxproj's
churn becoming a real merge burden — the moment maintaining it by hand costs more than pinning a
generator, this ADR is reopened with that evidence in hand.

**Deferral.** Recorded because the raise names it: the demo could conceivably be produced some
other way this month. What unblocks it: nothing — the measurement rungs still require an
installable app, so deferral moves the date and buys nothing.

## What this ADR does NOT decide

- No App Store metadata, no marketing surface, no icons beyond what the build demands. Each is
  later work with its own justification.
- No entitlements and no capabilities. The app reads photos through `PhotosPicker`, which runs
  out-of-process and needs no usage string; the `Info.plist` is fully generated.
- No CI device lane. Rung 31 owns CI; the shared scheme is what makes that lane possible later.
  Until then the shell build is a manual landing gate (ROADMAP, Harness backlog).
- No bench-grade measurement. A first device run's numbers are hand-run observations with device
  and OS attached — the measurement rungs own the ratified path, and no hand-run number is
  dressed as theirs.
- The eval-CLI boundary (the README's "offline tooling over the export, not code here") stays as
  it is; its revision is its own ADR when real data exists to evaluate.

## Consequences

- A new top-level `App/` directory: the project, the shim, the xcconfig pair (one committed, one
  git-ignored). `Package.swift` is untouched; the suite and its scheme are untouched — proven at
  the feat landing by running `test-clean` with the project present in the tree.
- `make bootstrap` remains the first command: the folder reference bundles whatever
  `Sources/InferlensApp/Models` holds, and it holds the verified bytes only after the fetch
  (invariant 8, unchanged).
- The manual device checkpoint becomes possible: install, classify, thumbs, export — the loop
  end to end on hardware, and the first ledger rows whose `DeviceIdentity` is a real phone
  (invariant 7's columns carrying device truth).
- The README's launchability sentences move to the new truth at this rung's docs commits, each
  anchored to its moment, none dressed as bench output.
